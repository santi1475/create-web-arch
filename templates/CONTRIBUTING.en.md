# Contributing Guide

## Commit Standard — Conventional Commits

### Format

```
<type>[optional scope]: <short description>

[optional body — explain WHY, not how]

[footer — issue refs, BREAKING CHANGE]
```

### Allowed types

| Type        | When to use                                          |
|-------------|------------------------------------------------------|
| `feat`      | New user-facing functionality                        |
| `fix`       | Bug fix                                              |
| `docs`      | Documentation only                                   |
| `style`     | Formatting, spaces, commas (no logic changes)        |
| `refactor`  | Refactor without new feature or bug fix              |
| `test`      | Add or fix tests                                     |
| `chore`     | Maintenance, dependencies, scripts                   |
| `perf`      | Performance improvements                             |
| `ci`        | CI/CD pipeline changes                               |
| `build`     | Build system, external tooling                       |
| `revert`    | Revert a previous commit                             |

### Examples

```bash
# Feature with scope
feat(auth): add Google OAuth login

# Bug fix referencing issue
fix(cart): prevent duplicate items on rapid click
Closes #234

# Breaking change
feat(api)!: change response envelope to JSON:API spec

BREAKING CHANGE: all API responses are now wrapped in { data, meta }.
Update all fetch calls accordingly.

# Chore
chore(deps): upgrade Next.js to 15.3
```

### Key rules

- Maximum **50 characters** in the title line
- Use **present imperative**: `add` not `added`, `fix` not `fixed`
- **No trailing period** on the title
- Separate body from title with a **blank line**
- Body explains the **what and why**, not the how

## Branch flow

```
main          ← production, protected, merge via PR only
develop       ← main integration branch
feature/xxx   ← new features  (branches from develop)
fix/xxx       ← bug fixes     (branches from develop)
hotfix/xxx    ← urgent fix    (branches from main)
release/x.x   ← release prep
```

### Branch naming

```bash
feature/user-authentication
feature/payment-gateway
fix/cart-duplicate-items
hotfix/critical-xss-vulnerability
release/2.1.0
```

---

## Pull Requests

1. PR title must follow Conventional Commits format
2. Describe what changes and why
3. Reference related issue (`Closes #123`)
4. At least 1 approved review before merge
5. Squash merge into `develop`, merge commit into `main`

## 🏗 Architecture & Code Generators

This project uses a precise folder structure generated by `create-web-arch`. To maintain architectural consistency across the team, **avoid creating files and folders manually**. 

Instead, always use the automated CLI generators:

```bash
# Components (Client/Server)
npx create-web-arch generate component MyComponent

# Custom Hooks
npx create-web-arch generate hook Auth

# Services / API integrations
npx create-web-arch generate service User

# Full Features (Only if using feature-based architecture)
npx create-web-arch generate feature Checkout
```

Using these commands ensures that your files are placed in the correct directories, aliases are correctly applied, and the project remains perfectly organized as it scales.
